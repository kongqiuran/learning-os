# Learning OS Step11-6 第一批真实用户测试方案

## 1. 测试目标

第一批测试不是证明“用户觉得 AI 很酷”，而是验证一条可收费闭环：

```text
目标学生出现
→ 愿意注册
→ 能独立上传真实课程资料
→ 能成功生成学习内容
→ 生成结果节省复习时间
→ 一周内再次使用或愿意付费
```

核心问题只有三个：

1. 学生能否在没有开发者指导的情况下完成首次价值体验？
2. 生成内容是否比自己整理资料更省时间、更适合考试？
3. 哪一种收费方式能让至少一部分学生真实做出支付选择？

## 2. 10名测试用户结构

不要全部来自熟人或同一专业。建议配额：

| 人数 | 用户类型 | 目的 |
|---:|---|---|
| 4 | 两周内有考试、资料较完整 | 验证最强刚需和付费窗口 |
| 2 | 理工科公式/计算型课程 | 验证公式、题型和长教材 |
| 2 | 文科/记忆型课程 | 验证长文本总结和章节结构 |
| 1 | 主要使用课件和笔记、无教材 | 验证缺少 TEXTBOOK 的降级体验 |
| 1 | 资料较乱或包含扫描 PDF | 暴露真实格式与 OCR 阻塞 |

筛选条件：当前确实有一门需要复习的课程；愿意使用自己的非敏感课程资料；同意完成 30–45 分钟首次测试和一周回访。

## 3. 用户来源与招募顺序

### 第一优先：可观察但非产品团队成员

- 同校不同专业同学；
- 课程群、考试复习群；
- 学生会、学习互助群、图书馆自习群；
- 已经在手工整理 PPT/真题的学生。

### 第二优先：半公开校园渠道

- 校园墙、社群管理员合作；
- 小红书/微信群“考试资料整理”主题招募；
- 给课程资料共享者提供免费测试名额。

### 招募文案原则

- 不承诺提分；
- 明确是测试版，可能失败；
- 强调“把教材、课件、笔记整理成复习内容”；
- 提醒不要上传含敏感个人信息或无权分享的资料；
- 测试补偿建议为小额饮品券或一个月早鸟权益，避免只吸引薅福利用户。

## 4. 测试执行流程

### 第0天：准备

1. 为每人分配匿名编号 U01–U10；
2. 记录专业、课程类型、考试日期、资料类型和文件数量；
3. 确认生产健康、模型余额、备份和故障联系人；
4. 准备一个不含真实隐私的演示课程，仅在用户完全卡住时使用。

### 第1天：首次任务（30–45分钟）

观察员只说任务，不教按钮位置：

> 请注册 Learning OS，为你最近要复习的一门课程建立学习空间，上传你认为重要的资料，生成学习内容，并用课程助手问一个真实问题。

记录：

- 每一步开始/完成时间；
- 用户说出的困惑原话；
- 观察员介入次数和原因；
- 上传文件类型、大小、成功/失败；
- 生成任务状态、耗时和失败阶段；
- 用户第一次认为“有用”的具体内容。

只有出现数据安全风险、连续卡住 3 分钟或准备放弃时才介入。

### 第2–6天：自然使用

不要求每天打卡，只发送一次提醒：在真实复习时继续使用。记录是否再次打开课程、追加资料、重新生成和使用助手。

### 第7天：回访与价格选择

进行 15 分钟访谈，完成 9 题问卷，并展示真实但不立即扣款的价格选择。重点记录用户愿意为哪种结果付费，而不是只问“你愿不愿意”。

## 5. 漏斗与指标定义

| 指标 | 定义 | 首批通过线 |
|---|---|---:|
| 注册完成率 | 完成注册人数 / 打开注册页人数 | ≥ 90% |
| 首次激活率 | 24小时内创建课程+成功上传+生成完成人数 / 注册人数 | ≥ 60% |
| 上传成功率 | 成功保存文件数 / 提交文件数 | ≥ 90%（扫描件单列） |
| 生成启动率 | 点击生成的激活用户 / 成功上传用户 | ≥ 80% |
| 生成成功率 | completed 任务 / 已启动任务 | ≥ 85% |
| 首次价值时间 | 注册至首次 completed 的中位数 | ≤ 15分钟 |
| 无协助完成率 | 未经观察员介入完成激活人数 / 注册人数 | ≥ 50% |
| AI助手使用率 | 至少完成一次问答的激活用户 / 激活用户 | ≥ 30% |
| 7日复用率 | 第2–7天再次产生有效操作人数 / 激活用户 | ≥ 30% |
| 强付费信号 | 选择具体价格并愿意进入付费试用的人数 | ≥ 3/10 |

有效操作定义为上传新资料、成功生成/重新生成、查看两个以上内容章节或完成助手问答，不把单纯登录计为复用。

## 6. 当前埋点缺口与首批替代方案

系统目前只有课程、文档、LearningPackage 和 UsageRecord，不能直接得到全部漏斗。首批10人可使用匿名测试表人工记录，但不得长期依赖人工。

人工表字段：用户编号、注册时间、课程创建时间、首次上传、文件结果、生成开始、生成完成、失败阶段、助手使用、观察员介入、D7回访、价格选择。

进入50人 Beta 前应新增事件：register_completed、course_created、upload_started/completed/failed、generation_started/completed/failed、assistant_asked、legal_accepted、account_deleted。事件必须绑定内部 user_id，不记录资料正文。

## 7. 反馈问卷（9题）

1. 你今天整理的是哪类课程，距离考试还有多久？
2. 从注册到生成完成，哪一步最难理解？为什么？
3. 生成内容中最有用的部分是什么？请指出具体内容。
4. 哪些内容不准确、太泛或你不敢直接使用？
5. 与自己手工整理相比，它为你节省了多少时间？（没有/少于30分钟/30–60分钟/1–3小时/3小时以上）
6. 你是否会在下一门课程继续使用？（一定不会/可能不会/不确定/可能会/一定会）
7. 如果只能保留一个功能，你会保留资料整理、学习包、课程知识还是 AI 助手？
8. 你更愿意选择哪种价格：¥3.9 单门一次、¥9.9 月度基础、¥19.9 月度 Pro、只使用免费版、都不会？
9. 什么变化会让你愿意马上付费？

问卷后追加开放访谈，不要用“你觉得好不好”诱导正面答案。

## 8. 第一版收费验证

### 阶段A：10人可用性测试

- 全部免费，但明确展示未来价格选项；
- 每人当前 Free 额度保持有限；
- 不以赠品换取“愿意付费”的口头回答。

### 阶段B：20–30人价格测试

建议同时测试两个真实选择，而不是只问卷：

- 单门考试包：¥3.9，包含一次完整整理和一次重整；
- 早鸟月卡：¥9.9，包含合理的课程数和20次以内整理。

若真实支付接入尚未完成，可人工收款后由管理员配置额度，但必须记录退款规则和服务失败补偿。不要宣传“无限生成”。

### 阶段C：正式套餐候选

只有生成成功率达到 95%、P90 首次价值时间可控且至少 10% 激活用户真实付费后，再测试 ¥19.9–29.9 的 Pro 套餐。

## 9. 问题分级与响应

| 等级 | 定义 | 处理 |
|---|---|---|
| P0 | 数据串用户、账号误删、资料泄露 | 立即停止测试并保全日志 |
| P1 | 无法注册/上传/生成，且无绕行方案 | 当日修复，受影响用户补额度 |
| P2 | 文案不清、需要观察员提示、结果部分错误 | 进入下一迭代，统计出现频率 |
| P3 | 样式、偏好和低频建议 | 记录但不打断验证 |

每天结束形成：完成用户数、激活数、失败任务、P0/P1、主要困惑原话、第二天处理决定。

## 10. Go / Iterate / Stop 决策

### Go：扩大到50人

- 无 P0；
- 激活率 ≥ 60%；
- 生成成功率 ≥ 85%；
- 至少 6/10 能指出具体有用内容；
- 至少 3/10 给出强付费信号。

### Iterate：再做一轮10人

- 用户认可结果，但卡在上传、等待或引导；
- 生成成功率 60%–85%；
- 付费意愿存在但价值表述不清。

### Stop / 改方向

- 多数用户即使成功生成也找不到具体价值；
- 真实资料普遍无法解析；
- 用户只愿意免费尝鲜且不愿再次使用；
- 成本和人工支持随用户线性增长且无法通过产品解决。

## 11. 本轮最重要的学习目标

第一批10人结束时，团队必须能用一句可验证的话回答：

> 哪一类学生，在什么时间点，上传什么资料后，会因为 Learning OS 帮他节省了什么具体工作，而愿意支付多少钱？

如果还只能回答“大学生觉得 AI 学习工具不错”，则测试没有完成商业验证。
