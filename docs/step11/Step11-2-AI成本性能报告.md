# Learning OS Step11-2 AI 成本与性能报告

## 1. 结论摘要

当前首次生成的模型调用次数为：

```text
未缓存 DocumentAnalysis 的文档数 N
+ 1 次 CourseAnalyzer
+ 1 次 LearningPackageGenerator
= N + 2 次
```

重新生成时，已有 DocumentAnalysis 会复用，通常仍需 **2 次模型调用**。系统目前没有保存 API 返回的 token usage、缓存命中或单次费用，因此下文是基于代码、字符上限和官方单价的工程估算，不是真实账单。

生产配置使用 `deepseek-v4-flash`。DeepSeek 官方当前标价为每百万 token：缓存命中输入 0.0028 美元、缓存未命中输入 0.14 美元、输出 0.28 美元。官方也说明中文字符通常约为 0.6 token，最终计费以 API usage 为准。

官方资料：

- [DeepSeek Models & Pricing](https://api-docs.deepseek.com/quick_start/pricing)
- [DeepSeek Token & Token Usage](https://api-docs.deepseek.com/quick_start/token_usage/)
- [DeepSeek Context Caching](https://api-docs.deepseek.com/guides/kv_cache)

## 2. 当前调用链

| 阶段 | 调用次数 | System Prompt | User Prompt 主要内容 | 当前限制 |
|---|---:|---:|---|---|
| DocumentAnalyzer | 每个未分析文档 1 次 | 1,388 字符 | document_type、source_type、提取文本 | 每个文档截断到 60,000 字符 |
| CourseAnalyzer | 每次生成 1 次 | 1,542 字符 | 所有文档分析 JSON 和权重 | 没有课程级总长度限制 |
| LearningPackageGenerator | 每次生成 1 次 | 2,793 字符 | 完整课程分析 JSON | 没有输入/输出 token 上限 |

流程为串行执行：文件解析 → 文档1模型 → 文档2模型 → … → 课程模型 → 学习包模型。模型网络重试最多 3 次，间隔为 2、5、10 秒；JSON 解析失败时会把错误输出追加到下一轮上下文，重试成本可能明显高于首轮。

## 3. Token 与费用估算

估算口径：中文正文按 0.6 token/字符；缓存命中不作为保守预算；DocumentAnalysis 输出按每份约 2,000 token；CourseAnalysis 约 4,000–6,000 token；最终学习包约 8,000–10,000 token。页面数只用于推导典型文字量，真实教材排版会造成较大差异。

| 场景 | 假设 | 调用数 | 输入 token 估算 | 输出 token 估算 | 单次费用估算 |
|---|---|---:|---:|---:|---:|
| 小课程，10页内 | 1份资料，约5,000中文字符 | 3 | 约 12,000 | 约 14,000 | 约 $0.0056 |
| 普通课程，约50页 | 3份资料，合计约25,000字符 | 5 | 约 31,000 | 约 19,000 | 约 $0.0097 |
| 大型课程，200页以上 | 5份资料，合计约100,000字符 | 7 | 约 92,000 | 约 26,000 | 约 $0.0202 |
| 极端文件堆积 | 10份文件，每份都达到60,000字符截断 | 12 | 约 390,000+ | 约 35,000+ | 约 $0.064+ |

计算公式：

```text
费用 = 缓存未命中输入 token × $0.14 / 1,000,000
     + 缓存命中输入 token × $0.0028 / 1,000,000
     + 输出 token × $0.28 / 1,000,000
```

如果某一阶段连续产生无效 JSON 或网络失败，最坏会尝试 3 次；不能简单按 3 倍估算，因为 JSON 重试会额外携带上一轮错误输出，但建议商业预算至少预留正常成本的 **2.5–3 倍异常系数**。

## 4. 性能分析

### 文件解析

- PDF 使用 PyMuPDF 逐页提取，PPTX 遍历文本形状，TXT/MD 整体读取；正常文字文件通常远快于模型调用。
- 扫描 PDF 无 OCR，会在生成阶段失败。
- 上传时整个文件一次性读入内存，单文件上限 50MB；并发大文件会放大内存压力。

### 模型等待

- 文档分析完全串行，是课程资料较多时的主要耗时来源。
- 每次请求超时 120 秒，最多 3 次尝试。单阶段在极端网络情况下可能持续约 6 分钟，N 个文档会线性累加。
- 非流式调用，用户只能轮询数据库状态，不能看到模型输出进度。

### 数据库与后台任务

- 每个进度阶段都会单独开启数据库 Session 并更新 LearningPackage；操作本身轻量，但产生较多短事务。
- DocumentAnalysis 会复用，避免重复文档成本；但没有内容哈希和分析器版本，文件或 Prompt 规则升级后无法判断缓存是否过期。
- FastAPI BackgroundTasks 不持久化；部署、崩溃或多实例切换会丢任务。

## 5. 成本爆炸点与风险等级

| 风险 | 等级 | 说明 |
|---|---|---|
| 没有记录 token usage 和实际费用 | High | 无法验证毛利、异常重试成本和用户真实消耗 |
| 课程文档数量、总存储和课程总输入无额度 | High | 单用户可上传大量 50MB 文件并触发 N+2 次调用 |
| 文档串行分析 | High | 大课程耗时随文档数线性增长 |
| CourseAnalyzer 输入无总上限 | High | 文档分析 JSON 可持续膨胀，最终达到上下文或延迟风险 |
| 未设置 response_format=json_object | Medium | Prompt 要求 JSON，但未启用模型原生 JSON Output，增加解析重试概率 |
| 未设置 max_tokens | Medium | 输出成本和响应时长缺少硬边界 |
| 生成任务非持久队列 | High | 用户付费后仍可能因部署丢失任务 |
| 缓存无法按内容版本失效 | Medium | Prompt 升级后旧 DocumentAnalysis 继续复用，结果不一致 |

## 6. 商业定价建议

模型直接成本非常低，当前主要成本不是 token，而是失败重试、存储、运维、支持和不可控大文件。建议不要按“每 token 加价”销售，而按可理解的学习任务额度定价。

### Beta 建议

- Free：每月 3 次课程整理，限制课程数、单课程文档数和总存储；保留当前体验。
- Pro 早鸟：建议验证区间 **¥9.9–19.9/月**，每月 20–50 次整理，先测试付费意愿，不承诺无限量。
- 单次包：可测试 **¥2.9–4.9/次完整课程整理**，适合考试周用户。

即使按普通课程正常成本约 $0.01、异常预算 $0.03 估算，模型毛利空间仍充足；真正需要先验证的是完成率和用户愿意为“节省复习时间”支付多少，而不是追求最低 token 单价。

## 7. 优化优先级

1. **P0：记录真实 usage**：每次模型调用保存 stage、model、prompt/cache/output token、耗时、重试、user_id、course_id 和 package_id。
2. **P0：增加课程级硬限制**：文档数、总提取字符、单课程存储、并发任务数。
3. **P0：持久任务队列与超时恢复**：避免付费任务因进程重启丢失。
4. **P1：三阶段聚合架构**：按教材、课堂资料、考试资料分阶段增量分析。
5. **P1：启用原生 JSON Output 并设置 max_tokens**：降低解析失败和输出失控。
6. **P1：有限并行文档分析**：每课程 2–3 并发，并设置账号级并发上限。
7. **P2：内容哈希与 Prompt 版本缓存**：只重算变化阶段。

## 8. 上线前必须补的观测指标

- 单次完整生成 P50/P90/P99 总耗时；
- 各 stage 的输入、缓存命中、输出 token 和费用；
- 文档解析失败率、JSON 重试率、模型网络重试率；
- 每门课程文档数、总字符数和总存储；
- 用户点击生成到 completed 的转化率；
- 失败后重试成功率和任务卡死数量。
